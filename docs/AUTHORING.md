# 創作問題 執筆・品質保証要件 v1(CAR形式)

執筆: Fable / ブラインドテスト・監修: Sonnet / 最終検品: Fable+ユーザー

## 1. 問題形式(CARフレームワーク)

各問題の本文(passage)は **CAR構造のミニストーリー**とし、最後に登場人物の心情を問う。

- **C: Context(状況)** 1〜2文 — 登場人物、関係性、場面の背景
- **A: Action(行動・出来事)** 1〜2文 — 何が起きたか、何をしたか
- **R: Result(結果・反応)** 1〜2文 — その後の様子。発話・仕草・行動で感情を「見せる」
- 分量: 120〜250字。現代の生活場面(職場・家族・友人・学校・近所など)
- 設問はアプリ共通の「この人物の感情を表す一言を」

## 2. 執筆ルール

1. ターゲット感情語(◎)は `data/emotion_lexicon.json` から選ぶ
2. **本文にターゲット語・その活用形・直接的な同義語を書かない**(感情は描写で示す)
3. 難易度: ★1 感情が明白 / ★2 近い語が競る(◎と○の選び分けが問題の本体)/
   ★3 言葉や行動と感情がズレる(強がり・建前・照れ隠し)
4. ○(許容)1〜2語・△(ズレ)1〜2語は「系統は近いが精度が違う」勾配で設計し、
   why にその差を書く
5. cues は「本文の表現 → 読み取れること」形式で2〜3個
6. 登場人物・場面は問題間で偏らせない(性別・年代・立場を分散)

## 3. データスキーマ(アプリ互換)

```json
{
  "id": "car001", "category": "哀", "difficulty": 2,
  "passage": "(CARミニストーリー)",
  "answer": { "word": "悔しい" },
  "ok": [{ "word": "…", "why": "…" }],
  "poor": [{ "word": "…", "why": "…" }],
  "cues": ["「…」→ …"], "explanation": "…",
  "source": { "work": "創作シナリオ", "author": "", "origin": "オリジナル" }
}
```

## 4. 品質ゲート: ブラインドテスト

「解けることが検証済みの問題だけを収録する」ための機械的ゲート。

- ソルバー(Sonnet)には **passage と感情語彙リスト(約60語)だけ**を渡す
  (選択肢・正解・解説は渡さない)
- ソルバーは語彙リストから心情に近い語を **順位付きで2つ** 挙げる
- **独立した3ソルバーのうち2以上の top-2 に◎語または○語が含まれれば PASS**
  (アプリの採点で○=許容である以上、読者が許容圏に届く問題は成立している。
  当初は◎のみを合格としたが、語彙リスト内の同義語衝突——「怖い/恐ろしい/恐怖」等——で
  文脈を正しく読めた解答が不合格になったため 2026-07-03 に修正。
  表記差の同族語(苛立ち/苛立たしい)は同一視する)
- 補助シグナル: ◎に一度も到達されなかった問題は、○リストにソルバーの選んだ同族語・許容語が
  欠けていないかを執筆者が点検し、正当なら○に追加する(ソルバーの近接誤答は許容圏の実測値)
- FAIL → 執筆者が文脈を強化して再テスト、または破棄
- 既存の小説モード50問にも同一プロトコルを適用し、PASS のみ「読解モード」として残す
- 実績(2026-07-03): 創作64問 → 55 PASS+9精査(7は○到達済みで許容圏を拡充、2は本文修正+再テスト)/
  小説50問 → 45 PASS、5問(n014, n021, n029, n043, n049)を文脈不足で削除

## 5. ブラインドテスト通過後の監修(Sonnet)

docs/CURATION.md の基準を CAR 用に読み替えて適用:
CAR構造になっているか / ターゲット語や同義語の本文露出がないか /
○△の勾配が成立しているか / 不自然な日本語がないか。
指摘は修正提案として返し、採否は Fable が判断する。
