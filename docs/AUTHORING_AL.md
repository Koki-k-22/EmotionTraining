# Active Listening 問題 執筆・品質保証要件 v1(AL形式)

執筆: Fable / ブラインドテスト・監修: Sonnet / 最終検品: Fable+ユーザー
計画の全体像は docs/PLAN_ACTIVE_LISTENING.md、理論的裏付けは docs/research/ を参照。

## 1. 問題形式

会話形式。**相手の発話に、気持ちを言葉にして返す**。

- **context**: 状況説明 1〜3文(誰が・どんな関係で・何があったか)。三人称で「あなた」の視点から
- **utterance**: 相手の発話(一人称)。必要なら behavior(様子)を「…と言って、〜した」の形で context 側に置いてよい
- 設問はアプリ共通: 「相手の気持ちを言葉にして返すなら?」(自由記述)
- 分量: context+utterance で 100〜220字。現代の生活場面(職場/家族/友人/パートナー)

## 2. 執筆手順(この順を守る)

### Step 1: 設計シートを先に書く

本文より先に、内部設計シート(コメントとして draft に残す)を埋める:

```
出来事:        何があったか
認識・評価:     本人がそれをどう受け取ったか(deep_lexicon の評価型から)
一次感情:       その受け取り方が生む感情(表層の下)
ニーズ:        本当はどうしてほしかったか
発話時点の心情:  ★必須1行。いま(発話の瞬間)支配的な心情。物語の中間感情と混同しない
表層:          発話に表れる感情(★2では感情語として明示)
```

### Step 2: 手がかりを2個以上、異なるチャネルで埋め込む

- 発話内の理由句(「〜なのに」「〜までして」「せっかく〜」)
- 発話の形式(繰り返し・言い直し・突き放し・自嘲)
- 状況・行動(context 側: 直前の出来事、様子)
- 手がかりはすべて同じ深層心情に収束すること。単一の感情語だけで解ける問題にしない

### Step 3: 正解セットを組む

- **deep.best(◎)を2〜3個**: 層の違う正解を認める(評価型+ニーズ型など)。各 phrase に
  keys(OR グループの AND、語幹で書く)と why(本文中の根拠)を付ける。
  ★1 も「深層の核は1系統」だが、正解セットとしての best は言い換え・隣接層を含め複数持ってよい
  (v1 のブラインドテストで、妥当な読みが複数層に分かれることが実証されたため)
- **surface(○)**: 表層感情の語+synonyms。「言い換えは間違いではないが浅い」の位置
- **deep.poor(△)を2個以上**: 異なる誤りタイプを2つ以上含め、少なくとも一方は
  **根拠のない読み込み**(過剰解釈=性格・過去・第三者の意図の推定/本文にない要素)とする。
  もう一方は**ズレ**(対象違い/因果すり替え/程度の逸脱/表面固着/一般論への逃避/**時点のずれ**)
  から選ぶ。why にパターン名を書く(誤答パターン集: docs/research/research2_kokugo.md §3.2)
- **keys の安全規則**: 1文字のひらがな語幹(「ず」「か」等)は禁止(部分一致で誤爆する)。
  肯定・否定の両文脈で使われる語(「敬意」「信頼」等)は、単独グループにせず否定側の語幹と AND にする

## 3. 執筆ルール

1. ターゲットの深層心情は `data/deep_lexicon.json` から選ぶ(完全一致でなくてよいが類型は揃える)
2. **★2 では表層感情語を発話に明示する**(「ムカつく」「最悪」「もうやだ」等)。
   深層のフレーズそのもの(「軽く扱われた」等)は本文に書かない
3. **時点の固定**: ◎は常に**発話時点**の心情。発話が時間経過を語る場合
   (「不安だったけど見つかってほっとした」)、通過済みの中間感情(不安)を◎にしない。
   中間感情への言及が共感として自然な場合は○に置き、why に「時点」の注記を書く
4. 難易度:
   - ★1: 表層感情語なし。深層が1つで手がかり明示的。深層に触れれば◎
   - ★2: 表層感情語が発話に明示。◎=その下の深層、○=表層の言い換え。**本形式の本体**
   - ★3: 発話と感情がズレる(建前・強がり・自嘲・「別にいいけど」)+複合感情。
     ◎=言外の本音、○=発話の表面上の感情への丁寧な応答、△=字義通りの受け取り(表面固着)
5. modelReply は「実際に返す一言」: 仮説マーカー(「〜んですね」「〜のかもしれませんね」)を含み、
   断定・助言・説教・評価・自分語りを含まない。25〜60字。
   文型テンプレは docs/research/research3_training.md §10
6. cues は「本文の表現 → 読み取れる深層」形式で2〜3個
7. 話者の性別・年代・立場・カテゴリを問題間で分散。「あなた」との関係(同僚/親/子/友人/恋人等)を context で明示
8. 説明(explanation)には「なぜ深層を返すと相手は理解されたと感じるか」を1文入れる

## 4. データスキーマ

```json
{
  "id": "al001",
  "category": "怒",
  "difficulty": 2,
  "context": "同期の友人と帰り道。彼は昼の定例会議で、半月かけて準備した企画案を課長に途中で遮られ、「前にも言ったけど筋が悪い」と一言で済まされた。",
  "utterance": "途中まで言いかけたのに「筋が悪い」で終わりだよ? こっちは半月かけて準備したのにさ。マジでムカつくわ。",
  "surface": { "word": "腹立たしい", "synonyms": ["ムカつく", "怒り", "苛立ち", "腹が立つ"] },
  "deep": {
    "best": [
      {
        "phrase": "半月の努力ごと軽く扱われたと感じた",
        "keys": [["努力", "半月", "準備", "頑張"], ["軽く", "無駄", "踏みにじ", "無下", "雑に"]],
        "type": "評価:軽視",
        "why": "「半月かけて準備したのに」が核。怒りの対象は結論ではなく扱いの軽さ"
      }
    ],
    "poor": [
      { "phrase": "課長に嫌われていると感じた", "why": "過剰解釈: 人間関係の悪さを示す根拠が本文にない" }
    ]
  },
  "cues": ["「半月かけて準備したのに」→ 怒りの核は結論ではなく、努力への扱い"],
  "modelReply": "半月かけたのに、中身も聞いてもらえなかった感じがして悔しいよね。",
  "explanation": "…(深層を返すと相手は「怒っている人」ではなく「傷ついた人」として理解されたと感じる)",
  "source": { "work": "創作シナリオ", "author": "", "origin": "オリジナル" }
}
```

- `category` は◎(深層)の感情系統の10分類(既存互換、成績画面用)。★3 では表層でなく本音側で付ける
- keys の判定仕様: 各 OR グループのいずれかの語幹を含む AND がすべて成立で一致(NFKC 正規化・空白除去後の部分一致)

## 5. 品質ゲート: ブラインドテスト v2

「解ける+根拠が本文にある」問題だけを収録するための機械的ゲート。

- ソルバー(Sonnet×3、視点別ペルソナ)に **context+utterance+深層心情リスト**
  (deep_lexicon の phrase 一覧、正解・選択肢・解説は渡さない)を渡す
- 指示: 「この人の**発話時点**の心情の核に最も近いものを順位付きで2つ選び、
  それぞれ**本文中の根拠**(引用)を添えよ」
- **合格: 3ソルバー中2以上の top-2 に、◎いずれかと同型(deep_lexicon の類型・意味が一致)の項目が含まれる**
- 表層感情のみを選んだソルバーが多数 → 手がかり不足。本文を増強して再テスト
- ソルバーが根拠を本文から引用できない◎ → 作問者の思い込み。正解セットか本文を修正
- ソルバーが通過済みの中間感情を選んだ → 時点設計の欠陥。本文か正解セットを修正
- FAIL → 修正して再テスト、または破棄

実績(2026-07-11): 試作10問 → 9 PASS+1修正(al003、ソルバー実測で許容圏を拡張)。
本執筆30問 → 30 PASS(al019/al022/al029 は実測に基づき許容圏を拡張)。
Sonnet監修で major 2(al038 勾配消失・al025 深層の本文露出)+ minor 12 を検出、反映済み。
既知の課題(v2 で対応): 「間接的に知らされる/外される」系の筋立てが6問(al007/019/020/023/024/029)で
重複感あり。次回拡充時は手がかりのバリエーション(態度の変化・時間の使い方等)を増やす。

## 6. ブラインドテスト通過後の監修(Sonnet)

以下を点検し、修正提案として返す(採否は Fable が判断):

- 設計シートの連鎖(出来事→評価→感情→ニーズ)が本文から復元可能か
- 深層フレーズ・その直接的な言い換えが本文に露出していないか(★2)
- ◎○△の勾配が成立しているか(○が◎と同じ深さになっていないか)
- △の「過剰解釈」が本当に本文から支持されないか(実は妥当なら◎/○へ移す)
- **時点の整合**: ◎が発話時点の心情か。中間感情が◎に紛れていないか
- keys が誤爆しないか(短すぎる語幹、否定文での誤マッチ)
- modelReply に断定・助言が混ざっていないか
- 不自然な日本語・設定の偏りがないか
